## low: Escrow amount formatting loses precision by converting bigint wei values to Number

id: fnd_sig-feat-library-12a33eb16e-6453_6a87addb74
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source frontend/src/app/[locale]/dashboard (feat_library_12a33eb16e)

evidence:
- frontend/src/app/[locale]/dashboard/page.tsx:146 (DashboardPage)
- frontend/package.json:27

totalAmount is a bigint from the chain, likely denominated in 18-decimal base units. Converting bigint wei values to JavaScript Number loses integer precision for ordinary token amounts above Number.MAX_SAFE_INTEGER base units, so the displayed BDAG amount can be rounded incorrectly before toFixed is applied. The project already depends on viem, which provides precise formatting utilities for bigint token values.

recommendation:
Use viem's formatEther or formatUnits on the bigint, then apply display rounding to the decimal string without converting the raw base-unit bigint to Number.

test analysis:
No linked tests are included, and there is no assertion covering large or fractional bigint token amount display.

suggested regression test:
Render a card with a mocked totalAmount of 1234567890123456789n and assert the formatted BDAG value is derived from exact bigint decimal formatting.

minimum fix scope:
frontend/src/app/[locale]/dashboard/page.tsx: replace Number(e.totalAmount) / 1e18 with a precise bigint formatting path.

repro:
Render a dashboard card with totalAmount set to a bigint value that is not exactly representable as a Number, such as 1234567890123456789n. The value is rounded during Number(...) conversion before display.

status: open

recommendation:
Include the escrow state field from the factory result or fetch each escrow's state, convert it with stateToStatus, and pass that value to StatusBadge. Use role information separately from lifecycle status if both are needed.

test analysis:
No linked tests are included for dashboard rendering, and there is no fixture asserting that complete/disputed chain states are displayed correctly.

suggested regression test:
Mock useReadContracts with an escrow record in complete and disputed states, render DashboardPage with a connected depositor and beneficiary wallet, and assert the corresponding StatusBadge status is complete/disputed rather than active/pending.

minimum fix scope:
frontend/src/app/[locale]/dashboard/page.tsx: preserve the escrow state from the contract result and use it when rendering the badge.

repro:
Open the dashboard with a wallet that is the depositor on an escrow whose contract/factory state is complete or disputed. The card badge still renders active because the badge condition only checks depositor address equality.

## medium: Tabs orientation is used for styling but not passed to Base UI root

id: fnd_sig-feat-library-1e5cfe145b-2336_140eefd792
category: api-contract
confidence: high
triage: contract-mismatch
status: open
feature: Node source frontend/src/components/ui#2 (feat_library_1e5cfe145b)

evidence:
- frontend/src/components/ui/tabs.tsx:8-22 (Tabs)

The component accepts `orientation` from `TabsPrimitive.Root.Props` and writes it to a custom `data-orientation` attribute, but because `orientation` is destructured out of `props`, it is never forwarded to `TabsPrimitive.Root`. Callers using `orientation="vertical"` will get vertical-looking styling while the underlying Base UI tabs root still receives its default orientation, which can break expected keyboard navigation and accessibility semantics.

recommendation:
Pass `orientation={orientation}` to `TabsPrimitive.Root` in addition to the styling data attribute.

test analysis:
No linked tests are included for this feature, and the package manifest exposes no test command. A visual-only snapshot would also miss this unless it checks primitive props or keyboard behavior.

suggested regression test:
Add a component test that renders `Tabs` with `orientation="vertical"` and verifies vertical keyboard behavior or that the primitive receives the vertical orientation attribute/semantics expected by Base UI.

minimum fix scope:
Update `frontend/src/components/ui/tabs.tsx` so `TabsPrimitive.Root` receives the destructured `orientation` prop.

repro:
Render `<Tabs orientation="vertical">` with multiple triggers, then inspect the underlying Base UI root props/behavior: the primitive is not given `orientation="vertical"` even though the wrapper applies `data-orientation="vertical"`.

## medium: Support reports read escrow state from the default chain instead of the connected chain

category: bug
confidence: medium

status: open

evidence:

- frontend/src/components/SupportModal.tsx:65-72 (handleSubmit)
- frontend/src/components/SupportModal.tsx:156-158 (SupportModal)

SupportModal captures the connected chain ID and displays that network in the metadata preview, but the contract state lookup always uses a module-level public client created for DEFAULT_CHAIN_ID. recommendation:
Create the public client from `getChain(chainId)` at submit time, or skip contract-state capture unless the route's escrow address is known to belong to the selected chain. Include read failures in internal diagnostics rather than silently conflating wrong-chain reads with milestone contracts.

test analysis:

suggested regression test:
Mock `useChainId` to a non-default supported chain, mock `createPublicClient`/`readContract`, submit the modal, and assert the read uses that chain's RPC config and the resulting payload contains the correct contractState.

minimum fix scope:
frontend/src/components/SupportModal.tsx

repro:
Connect to a supported non-default chain, open an escrow route for a contract deployed only on that chain, and submit a support report. The preview shows the connected network, but the state read is attempted against the default chain RPC and is silently ignored or reports the wrong state.

## medium: Unsupported-chain overlay leaves underlying controls keyboard-accessible

id: fnd_sig-feat-library-2a3675604c-6eec_f27045d9aa
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source frontend/src/components#1 (feat_library_2a3675604c)

evidence:
- frontend/src/components/chain-guard.tsx:64-67 (ChainGuard)

When the wallet is connected to an unsupported chain, ChainGuard visually dims the protected page and blocks pointer events, but it still renders the original focusable controls. `aria-hidden` only hides the subtree from assistive technology; it does not remove links, buttons, or form fields from sequential keyboard focus. A keyboard user can tab into and activate protected page actions while the UI indicates the page is blocked for an unsupported network.

recommendation:
Use the `inert` attribute on the dimmed content, conditionally render a non-interactive placeholder instead of children, or recursively disable known actionable descendants while unsupported. Keep the warning and switch buttons outside the inert region.

test analysis:
No tests were included for keyboard navigation or unsupported-chain behavior.

suggested regression test:
Render ChainGuard with a connected unsupported chain and a child button, simulate keyboard tab/activation, and assert the child action is not invoked while the chain-switch buttons remain usable.

minimum fix scope:
frontend/src/components/chain-guard.tsx

repro:
Connect on an unsupported chain, load a page wrapped by ChainGuard that contains a button or link, press Tab until focus reaches an underlying control, then press Enter or Space. The underlying control can still be activated despite the unsupported-chain state.

## medium: Footer support action can open the wrong fixed modal

id: fnd_sig-feat-library-2a3675604c-fb22_15665a0e84
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source frontend/src/components#1 (feat_library_2a3675604c)

evidence:
- frontend/src/components/footer.tsx:111-116 (Footer)
- frontend/src/components/SupportButton.tsx:15-19 (SupportButton)
- frontend/src/components/feedback-button.tsx:12-17 (FeedbackButton)

The footer's Support link does not target the support component directly. It clicks the first button in the document that has an aria-label and a className containing fixed. Both the support and feedback floating buttons match that predicate, so DOM order determines whether the Support footer link opens support or feedback. This is a user-visible correctness bug and will be brittle as more fixed aria-label buttons are added.

recommendation:
Replace the DOM query/click coupling with explicit state or a shared support-modal controller/context. At minimum, dispatch a named custom event or query a stable data attribute unique to the support button, not all fixed aria-label buttons.

test analysis:
No tests were included for footer interactions or coexistence of the floating support and feedback buttons.

suggested regression test:
Render Footer with both floating buttons present, click the localized Support footer item, and assert that SupportModal opens while FeedbackModal remains closed.

minimum fix scope:
frontend/src/components/footer.tsx and the component or shared state that owns SupportModal opening.

repro:
Render a page with both FeedbackButton and SupportButton where FeedbackButton appears earlier in DOM order, then click Footer -> Support. The feedback modal opens instead of the support modal.

## medium: Receipt subtracts the platform fee from an amount documented as already net

id: fnd_sig-feat-library-3908f5cfb5-c3bc_14c6260edf
category: bug
confidence: medium
triage: risk
status: open
feature: Node source frontend/src/lib (feat_library_3908f5cfb5)

evidence:
- frontend/src/lib/generateReceipt.ts:15 (ReceiptData.amount)
- frontend/src/lib/generateReceipt.ts:154-159 (generateReceipt)

The public input contract says `amount` is the net escrow amount in wei, but the receipt generator computes a new 0.5% fee from that value and displays `data.amount - protocolFee` as the net amount to seller. For callers that follow the interface, receipts understate the seller amount and report a synthetic fee that was already excluded. This is financial-output corruption in the generated PDF, even if no on-chain state is changed.

recommendation:
Clarify the data contract and use one representation consistently. Either rename/document `amount` as gross escrow amount and require callers to pass gross, or treat it as net and do not subtract the platform fee again. If both values are needed, add explicit `grossAmount` and `protocolFee` fields.

test analysis:
No tests were included for receipt generation or for the `ReceiptData` amount contract.

suggested regression test:
Add a receipt-generation unit test with `amount = 1000n` documented as net and assert the rendered financial rows do not calculate `995n` as the seller amount, or update the interface to gross and assert the gross/fee/net rows match the chosen contract.

minimum fix scope:
frontend/src/lib/generateReceipt.ts

## medium: Environment-provided contract addresses are accepted without runtime validation

id: fnd_sig-feat-library-4c9beaa399-d02e_c90f6cb849
category: bug
confidence: medium
triage: risk
status: open
feature: Node source frontend/src/lib/contracts (feat_library_4c9beaa399)

evidence:
- frontend/src/lib/contracts/addresses.ts:11-13 (CONTRACT_ADDRESSES)

The helpers promise viem Address values, but the environment values are only type-cast. A non-empty typo such as NEXT_PUBLIC_FACTORY_ADDRESS=foo or an incorrectly sized 0x string bypasses the fallback and is returned as an Address. That pushes a deployment configuration error into later contract calls, where failures are harder to diagnose and may affect all callers for that chain.

recommendation:
Validate each configured address at module initialization or in getContractAddresses using viem's isAddress, and throw a clear configuration error naming the invalid variable and chain id. Consider treating empty strings as unset if that matches deployment expectations.

test analysis:
No tests are linked for this source group, and there is no visible coverage for malformed NEXT_PUBLIC_* address values.

suggested regression test:
Add a unit test that loads the address module with an invalid NEXT_PUBLIC_FACTORY_ADDRESS and asserts that getFactoryAddress(1404) throws a clear configuration error rather than returning the invalid string.

minimum fix scope:
frontend/src/lib/contracts/addresses.ts

repro:
Set NEXT_PUBLIC_FACTORY_ADDRESS=foo, build/run the frontend, and call getFactoryAddress(1404). It returns foo typed as Address instead of rejecting the invalid configuration or falling back.
