## low: Escrow amount formatting loses precision by converting bigint wei values to Number

id: fnd_sig-feat-library-12a33eb16e-6453_6a87addb74
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Node source frontend/src/app/[locale]/dashboard (feat_library_12a33eb16e)

evidence:
- frontend/src/app/[locale]/dashboard/page.tsx:146 (DashboardPage)
- frontend/package.json:27

totalAmount is a bigint from the chain, likely denominated in 18-decimal base units. Converting bigint wei values to JavaScript Number loses integer precision for ordinary token amounts above Number.MAX_SAFE_INTEGER base units, so the displayed BDAG amount can be rounded incorrectly before toFixed is applied. The project already depends on viem, which provides precise formatting utilities for bigint token values.

recommendation:
Use viem's formatEther or formatUnits on the bigint, then apply display rounding to the decimal string without converting the raw base-unit bigint to Number.

test analysis:
No linked tests are included, and there is no assertion covering large or fractional bigint token amount display.

suggested regression test:
Render a card with a mocked totalAmount of 1234567890123456789n and assert the formatted BDAG value is derived from exact bigint decimal formatting.

minimum fix scope:
frontend/src/app/[locale]/dashboard/page.tsx: replace Number(e.totalAmount) / 1e18 with a precise bigint formatting path.

repro:
Render a dashboard card with totalAmount set to a bigint value that is not exactly representable as a Number, such as 1234567890123456789n. The value is rounded during Number(...) conversion before display.
