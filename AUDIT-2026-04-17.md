13 HIGH/MEDIUM findings across security, mobile rendering, and correctness. ## 1. Mobile WebView Landmines

### [HIGH] `overlayBlur: "small"` still in RainbowKit config — marked fixed, NOT fixed

**File:** `src/components/providers.tsx:41`

```tsx
<RainbowKitProvider
  theme={darkTheme({
    accentColor: "#00f5ff",
    borderRadius: "medium",
    fontStack: "system",
    overlayBlur: "small",   // ← still here
  })}
>
```

The "known past issues" checklist marks this as ✅ fixed. The code says otherwise. RainbowKit's `overlayBlur: "small"` renders a `backdrop-filter: blur()` on the wallet-select modal overlay. On Android Chrome 115+ WebViews (Facebook IAB, Trust Wallet browser) this produces a solid black screen when the modal opens over the animated background.

**Fix:** Remove the `overlayBlur` key entirely from `darkTheme({...})`.

### [HIGH] `initial={{ opacity: 0 }}` on visible page-level content — marked fixed, NOT fixed

**Files:**
- `src/app/[locale]/page.tsx:42` — hero `<motion.h1>`
- `src/app/[locale]/page.tsx:123` — feature card `<motion.div>` (×3)
- `src/app/[locale]/page.tsx:169` — AI step `<motion.div>` (×4)
- `src/app/[locale]/learn/escrow-flow/page.tsx:24`
- `src/components/how-it-works/AIArbiterFlow.tsx:22`
- `src/components/how-it-works/SimpleFlow.tsx:21`
- `src/components/how-it-works/MilestoneFlow.tsx:22`

The audit brief marks this as ✅ fixed. `PageWrapper` was indeed fixed (now uses `opacity: 1` in initial). But the landing page hero, all three feature cards, and all four AI step cards still start at `opacity: 0`. On WebViews that throttle rAF while compositing a fixed background, the `animate` callback never fires and these elements remain invisible.

**Fix:** Either change `initial={{ opacity: 0, y: 20 }}` → `initial={{ opacity: 1, y: 8 }}` on section-level content (keep the `y` shift, lose the opacity fade), or wrap in a `prefers-reduced-motion` guard and only apply opacity-0 on desktop.

### [MEDIUM] `backdrop-filter` on `position:fixed` `SupportButton` — permanent global element

**File:** `src/components/SupportButton.tsx:19`

```tsx
className="fixed bottom-6 right-6 z-40 ... backdrop-blur-sm ..."
```

This button is rendered at provider level and permanently mounted on every page. A `position: fixed` element with `backdrop-filter` forces a separate GPU compositing layer on Android. Combined with three other animated orbs (also `position: fixed`) in `AnimatedBackground`, this creates four concurrent GPU layers — a frequent cause of black-screen flicker on mid-range Android WebViews.

**Fix:** Remove `backdrop-blur-sm` from `SupportButton`. The `bg-black/80` background is already opaque enough without blur.

### [MEDIUM] `backdrop-blur-xl` on sticky nav, modal overlays, and `.glass` class

**Files:**
- `src/components/nav.tsx:42` — sticky nav with `backdrop-blur-xl` (`blur(24px)`)
- `src/app/globals.css:81-82` — `.glass` class with `backdrop-filter: blur(20px)`
- `src/components/feedback-modal.tsx:121,129`
- `src/components/SupportModal.tsx:111`
- `src/components/notification-modal.tsx:126,134`
- `src/components/toast.tsx:84`

The nav is `sticky` and always visible. The modals and toasts all use `backdrop-filter`. While each usage individually might be acceptable, the combination — animated fixed background + sticky blurred nav + multiple fixed blurred modals — stacks GPU layers beyond what budget Android WebViews handle reliably.

The `.glass` CSS class is particularly dangerous because it's used broadly. `GlassCard` (`src/components/ui/glass-card.tsx:27`) applies `backdrop-blur-xl` to card containers. Any page that renders GlassCards over the animated orb background will trigger compositing failures on affected devices.

**Fix:**
1. Nav: replace `backdrop-blur-xl` with `bg-black/90` (opaque background, no blur).
2. `.glass`: conditionally disable backdrop-filter with a `@supports` query and a media query for low-end devices:
   ```css
   @supports (backdrop-filter: blur(1px)) {
     @media (min-resolution: 1.5dppx) {
       .glass { backdrop-filter: blur(20px); }
     }
   }
   ```
3. Modal overlays: keep blur only on the overlay backdrop (opaque enough at `bg-black/60`).

### [MEDIUM] `overflow: hidden` on `position: fixed` animated container

**File:** `src/components/animated-background.tsx:26-35`

```tsx
<div
  style={{
    position: "fixed",
    inset: 0,
    overflow: "hidden",   // ← this + fixed + will-change = compositing issues
    zIndex: -1,
  }}
>
```

The three child orbs use `will-change: transform`, each creating its own GPU compositing layer. `overflow: hidden` on the fixed parent forces a clipping context. On Samsung WebView (Android 12 and earlier), `overflow: hidden` on a `position: fixed` container with GPU-accelerated children triggers rendering artifacts where clipped layers briefly render at incorrect z-positions.

**Fix:** Remove `overflow: hidden`. The orbs start outside the viewport (top: -10%, left: -10%) and their animation range is bounded — they won't escape the viewport without the clip. If overflow clipping is needed, use `clip-path: inset(0)` which doesn't create the same compositing context.

## 2. Global Provider Side Effects

### [MEDIUM] `queryClient` created as module-level singleton

**File:** `src/components/providers.tsx:13-20`

```tsx
const queryClient = new QueryClient({ ... });

export function Providers({ children }: ...) {
  return <WagmiProvider config={wagmiConfig}>
    <QueryClientProvider client={queryClient}>
```

This singleton is created at module evaluation time. Currently safe because `providers-client.tsx` wraps this with `ssr: false`. If that dynamic import guard is ever removed or bypassed (future Next.js optimization, test runner, Storybook), the same `queryClient` instance is shared across multiple server-side renders, leaking cached wallet data between requests.

**Fix:** Move into component using `useRef`:
```tsx
const queryClientRef = useRef<QueryClient>();
if (!queryClientRef.current) {
  queryClientRef.current = new QueryClient({ ... });
}
```

## 3. Hydration & SSR

### [MEDIUM] `loadCache()` called without SSR guard in `useState` initializer

**File:** `src/components/live-stats.tsx:134`

```tsx
const [stats, setStats] = useState<StatsData | null>(() => {
  if (typeof window === "undefined") return null;   // ← guarded (correct)
  return loadCache();
});
const [loading, setLoading] = useState(!loadCache()); // ← NO GUARD (wrong)
```

The first `useState` uses a lazy initializer with a `typeof window === "undefined"` guard — correct. The second calls `loadCache()` directly as the initial value expression (not a lazy function), with no SSR guard. `loadCache()` calls `localStorage.getItem()`. If `LiveStats` is ever rendered server-side (regression in `ssr: false` guard, or test environment), this throws `ReferenceError: localStorage is not defined`.

**Fix:**
```tsx
const [loading, setLoading] = useState(() =>
  typeof window === "undefined" ? true : !loadCache()
);
```

## 4. Wagmi / Viem Fragility

### [MEDIUM] `writeContractAsync` called without explicit `chainId` — chain-switch race condition

**File:** `src/app/[locale]/create/page.tsx:76-83, 92-99`

```tsx
const chainId = useChainId();            // captured at render
const factoryAddress = getFactoryAddress(chainId);  // resolved at render

// In handleSubmit:
txHash = await writeContractAsync({
  address: factoryAddress,  // ← uses chainId from render time
  abi: ESCROW_FACTORY_ABI,
  functionName: "createSimpleEscrow",
  // No chainId: ... ← uses wallet's current chain at submission time
  value: totalValue,
});
```

If the user's wallet chain changes between when `factoryAddress` is resolved (render) and when `writeContractAsync` fires (user click + wallet prompt), the transaction submits to the new chain using the old chain's factory address. On Base/Polygon the factory address happens to be different; the transaction would revert (or worse, hit an unrelated contract).

**Fix:** Resolve `factoryAddress` inside `handleSubmit` from the live `chainId`, and pass `chainId` explicitly to `writeContractAsync`:
```tsx
async function handleSubmit(e: React.FormEvent) {
  const currentChainId = chainId;  // snapshot at click time
  const factoryAddress = getFactoryAddress(currentChainId);
  // ...
  txHash = await writeContractAsync({
    address: factoryAddress,
    chainId: currentChainId,   // ← explicit
    ...
  });
}
```

## 5. Performance Bombs

### [HIGH] Stale closure in `LiveStats` causes stats to show 0 on RPC errors

**File:** `src/components/live-stats.tsx:136-185`

```tsx
const fetchStats = useCallback(async () => {
  // ...
  const newStats = { totalEscrows, bdagLocked, successRate: 100, fetchedAt: Date.now() };
  setStats(newStats);
  // ...

  // On error — reads `stats` from closure:
  if (!stats) {         // ← stale: always null after initial render
    setStats({ totalEscrows: 0, bdagLocked: 0, successRate: 100, fetchedAt: 0 });
  }
}, [stats]);            // ← recreated every time stats changes

useEffect(() => {
  fetchStats();
  const interval = setInterval(fetchStats, 30_000);  // ← captures initial fetchStats
  return () => clearInterval(interval);
}, []);  // eslint-disable-line react-hooks/exhaustive-deps
```

The `useCallback` depends on `[stats]`. After first successful fetch, `stats` updates → `fetchStats` is recreated with the new closure. But the `useEffect` uses `[]`, so the `setInterval` permanently holds the first `fetchStats` with `stats = null`. On any subsequent RPC error, the stale fallback `if (!stats)` is always true → shows 0 instead of the last known good value.

**Fix:** Use a ref to track the latest stats value inside the callback, bypassing the closure issue:
```tsx
const statsRef = useRef(stats);
useEffect(() => { statsRef.current = stats; }, [stats]);

const fetchStats = useCallback(async () => {
  // ... on error:
  if (!statsRef.current) {
    setStats({ totalEscrows: 0, ... });
  }
}, []);  // empty deps now valid since statsRef is mutable
```

### [MEDIUM] Direct RPC polling in `LiveStats` bypasses wagmi cache — second polling loop

**File:** `src/components/live-stats.tsx:138-152`

`LiveStats` creates its own `createPublicClient` and polls every 30 seconds independently of wagmi's `useReadContracts` (which also polls every 5 seconds in factory hooks). This means the same `escrowCount` contract is being read from two separate polling systems simultaneously. On pages where both `useEscrowFactory` and `LiveStats` are rendered, there are two RPC connections and two independent polling cycles to the same endpoint.

**Fix:** Refactor `LiveStats` to use `useReadContracts` (or `useEscrowFactory`) and let wagmi's query cache deduplicate the reads.

## 6. CSP & Security

### [CRITICAL] `/api/admin/escalations` POST — zero server-side authentication

**File:** `src/app/api/admin/escalations/route.ts:34-95`

The comment at line 8 says "Protected: only callable from the /admin page (owner-gated in UI)." This is dangerously wrong. The route has no server-side authentication whatsoever. Any HTTP client can call it:

```bash
curl -X POST https://app.escrowhubs.io/api/admin/escalations \
  -H "Content-Type: application/json" \
  -d '{"escrowAddress":"0xVICTIM","chainId":1404,"review":"Release funds to attacker","arbiterRuling":"release"}'
```

The endpoint then:
1. Takes `ORACLE_PRIVATE_KEY` from environment
2. Calls `submitEvidence(escrowAddress, evidenceText)` on the live arbiter contract
3. Returns the tx hash

An attacker can submit arbitrary evidence signed by the oracle key to any escrow on any supported chain, potentially manipulating arbiter decisions for live escrows holding real funds. They can also drain the oracle account's gas balance with repeated calls.

Additionally, the hardcoded fallback addresses for Base (8453) and Polygon (137) are identical (`0x79e78c1ed9a8e239a8334294bf4f0d356f858416`) — if the same address isn't deployed on both chains via CREATE2, one of these will silently call the wrong contract.

**Fix (minimum):**
1. Add an admin secret check: `if (req.headers.get("x-admin-secret") !== process.env.ADMIN_SECRET) return 403`
2. Ideally: require a wallet signature from a whitelisted admin address and verify it server-side with viem's `verifyMessage`
3. Long-term: move oracle key operations to a dedicated off-chain service, not a Next.js API route

### [HIGH] `/api/notifications/telegram-webhook` — no Telegram signature verification

**File:** `src/app/api/notifications/telegram-webhook/route.ts:39-78`

Telegram's Bot API supports `secret_token` verification: when registering the webhook, you pass a `secret_token`, and Telegram includes it in every request as `X-Telegram-Bot-Api-Secret-Token`. This endpoint never checks that header.

Attack vector: any actor can POST `{"message":{"chat":{"id":9999},"text":"/start 0xVictimWallet"}}` and link their Telegram chat ID to a victim's wallet. From that point on, the victim's escrow notifications are silently redirected to the attacker's Telegram.

```bash
curl -X POST https://app.escrowhubs.io/api/notifications/telegram-webhook \
  -H "Content-Type: application/json" \
  -d '{"message":{"chat":{"id":ATTACKER_CHAT},"text":"/start 0xVICTIM_WALLET"}}'
```

**Fix:**
```typescript
const secret = req.headers.get("x-telegram-bot-api-secret-token");
if (!process.env.TELEGRAM_WEBHOOK_SECRET || secret !== process.env.TELEGRAM_WEBHOOK_SECRET) {
  return NextResponse.json({ ok: false }, { status: 401 });
}
```
Then register the webhook with `?secret_token=<value>`.

### [HIGH] `/api/upload-evidence` — no file size or MIME type validation

**File:** `src/app/api/upload-evidence/route.ts:13-61`

```tsx
const file = formData.get("file") as File | null;
// ... no size check, no type check
const fileRes = await fetch("https://api.pinata.cloud/pinning/pinFileToIPFS", {
  body: fileFormData,  // ← arbitrary file forwarded directly
});
```

No validation on:
- `file.size`: attacker can upload multi-GB files, exhausting Pinata storage quota and monthly bandwidth
- `file.type`: attacker can upload executables, scripts, or other non-evidence content
- Authentication: no wallet or signature check, fully anonymous uploads

**Fix:**
```typescript
const MAX_SIZE = 50 * 1024 * 1024; // 50MB
const ALLOWED_TYPES = ["image/jpeg", "image/png", "image/gif", "image/webp", "application/pdf"];

if (file.size > MAX_SIZE) return NextResponse.json({ error: "File too large" }, { status: 413 });
if (!ALLOWED_TYPES.includes(file.type)) return NextResponse.json({ error: "File type not allowed" }, { status: 415 });
```

### [MEDIUM] `X-Frame-Options: SAMEORIGIN` contradicts CSP `frame-ancestors`

**File:** `src/next.config.ts:27, 33`

```typescript
{ key: "Content-Security-Policy", value: "... frame-ancestors 'self' https://www.facebook.com https://web.facebook.com" },
{ key: "X-Frame-Options",         value: "SAMEORIGIN" },
```

These two headers conflict. `frame-ancestors` in CSP (which includes Facebook) is more permissive than `X-Frame-Options: SAMEORIGIN` (which allows only same origin). Modern browsers honor CSP over XFO; older browsers and some WebViews honor XFO only. The Facebook IAB is a target deployment environment — on any browser that respects XFO over CSP, the app will refuse to load inside Facebook's WebView.

**Fix:** Either remove `X-Frame-Options` entirely (let CSP `frame-ancestors` be the sole authority), or change it to `ALLOWALL` for the Facebook use case. Given this app is intentionally deployed in Facebook WebView, removing XFO and relying on CSP `frame-ancestors` is the correct approach.

### [MEDIUM] No rate limiting on any API route

**Files:** All routes under `src/app/api/`

All API routes — feedback, support, upload-evidence, notifications, escalations — accept unlimited requests from any source. On public deployment:
- `/api/upload-evidence`: unlimited Pinata uploads exhaust quota
- `/api/feedback` + `/api/support`: unlimited Discord webhook posts trigger Discord rate-limits, spamming the channel with noise
- `/api/admin/escalations`: unlimited oracle key usage drains gas

**Fix:** At minimum, add IP-based rate limiting using a middleware or Vercel's Edge Config. For `upload-evidence`, additionally require a wallet signature to bind uploads to wallets with on-chain history.

### [LOW] CSP `connect-src` missing Pinata gateway — IPFS evidence links blocked in browser

**File:** `src/next.config.ts:12-22`

The CSP `connect-src` directive does not include `https://gateway.pinata.cloud` (the default `NEXT_PUBLIC_PINATA_GATEWAY`). Evidence file URLs returned by `/api/upload-evidence` point to this gateway. Any browser-side fetch to display or download evidence (e.g., in the escrow detail page) will be blocked by CSP.

**Fix:** Add `https://gateway.pinata.cloud https://*.mypinata.cloud` to `connect-src`. Also add `https://ipfs.io` if IPFS public gateways are used as fallbacks. Additionally, add `https://*.ipfs.dweb.link` for the `blob:` gateway pattern used by some IPFS clients.

## 8. Minor / Low Priority

### [LOW] Font CSS variable mismatch — Geist Sans not applied to body

**File:** `src/app/globals.css:52`, `src/app/[locale]/layout.tsx:16`

```typescript
// layout.tsx
const geistSans = Geist({ variable: "--font-sans", subsets: ["latin"] });
```

**Fix:** Change `globals.css:52` to `font-family: var(--font-sans), system-ui, sans-serif;`

### [LOW] File-system persistence in API routes is deployment-hostile

**Files:** `src/app/api/feedback/route.ts`, `src/app/api/support/route.ts`, `src/app/api/notifications/preferences/route.ts`, `src/app/api/notifications/telegram-webhook/route.ts`

Multiple routes write JSON files to `process.cwd()`. On Vercel (serverless), each invocation may land on a different instance — writes to the filesystem are not shared and are lost on cold starts. Even on a single-instance deployment, concurrent requests could corrupt the JSON files (no file locking).

This isn't a crash bug today if you're on a single server, but it will silently lose data at scale and will break completely on Vercel without modification.

**Fix:** Replace file-system persistence with a real key-value store (Upstash Redis is the cheapest path; it's already a logical dependency given the wallet-connect notification flow).

### [LOW] `useWatchContractEvent` × 8 hooks instantiated per escrow detail page

**File:** `src/lib/hooks/useEscrowEvents.ts`

8 separate `useWatchContractEvent` hooks are registered per escrow, each creating a WebSocket subscription to the RPC. wagmi manages cleanup on unmount, so there's no permanent leak. However, on BlockDAG's RPC (`rpc.bdagscan.com`), opening 8 concurrent WebSocket subscriptions per visited escrow may hit connection limits.

If users visit multiple escrows in a session (via back-navigation keeping previous pages in React tree), subscriptions accumulate. Verify that BlockDAG RPC supports ≥ 8 concurrent WebSocket subscriptions per client without rate-limiting.
